# Chat / Realtime Layer Architecture Roadmap

## Executive summary

The chat layer is structurally sound but carries a cluster of hand-maintained invariants that should be compile-time facts, plus two real cost/correctness gaps. Five proposals were evaluated; all five had their load-bearing claims verified against the code with no fabrications. The dominant risk is **overlap and over-bundling, not over-engineering** — Proposals 1, 3, 4, and 5 converge on the same four warts (dual status field, five optimistic flags, the single-in-flight `pendingClientSubmissionIdRef`, and carrying `clientSubmissionId` on the error payload).

Recommendation: ship three independent quick wins now, then fund two strategic spines — **server-authoritative subscription (P2)** and **typed status state machine + optimistic overlay (P1, Changes B+C)** — with **typed error channels (P4)** sequenced for its server-enum alignment because three other proposals depend on it. Treat Proposal 5 not as a standalone epic but as a small optional follow-on (deleting the reducer's `dbMessages` copy) once P1 lands, because that copy already has no render consumer.

Hard sequencing constraint: `clientSubmissionId` is **not** on the runtime error payload today. Four proposals' final phases depend on threading it through the pinned `react-ag-ui@0.0.26` `onError` boundary. Land that single wire-contract change once and let the others consume it; do not let four proposals each attempt it. Every behavioral change must gate on the replay-based integration harness (`apps/www/test/integration/`), not unit tests alone — the optimistic-submit path just shipped to `main` as default and the resume-policy gate was recently fixed.

## Strategic bets

### Bet A — Proposal 2 (full): Server-authoritative subscription + bounded Redis

**Problem.** The live-subscription decision is gated on lagging client-derived state, which is structurally a gated-on-self deadlock. `runtime-resume-policy.ts:20` sets `historyMode = isAgentWorking ? "active-resume" : "idle-finalized"`, and `isAgentWorking` is the client reducer's own status projection. If that projection is stale-idle while the server has an active run, `unstable_resume` stays false, `startRun` is never called, and no SSE opens even though the server is streaming. The recently-fixed deadlock patched one mis-derivation; the structural coupling that produces the whole class is unchanged.

**Proposed change.** Move the liveness decision to the server. The server already computes terminal run status (`route.ts:427` `getAgentRunContextByRunId`, `:450/:466` `isTerminalAgentRunStatus`). Expose it as `runActive: boolean` + `activeRunId: string | null` on `AgUiHistoryMessagesResult` (which has no such field today), and gate `historyMode`/`unstable_resume` on `serverRunActive` instead of client `isAgentWorking`. `isAgentWorking` becomes a pure UI hint (spinner, composer disable) with zero transport authority. A connection opened against an already-finished run replays and closes immediately, so this does not revive idle always-on streams.

**Migration phases.**

| Phase    | Change                                                                                                                                                                                                                       | Reversible                                  |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| 0        | Observability: spans at the three live-tail swallows + SSE open/close; wire existing `StreamDiagnostics` counters to a real sink                                                                                             | Yes (additive)                              |
| 1        | Buffer bound: `MAXLEN ~N` + `EXPIRE` on the two durable + one ephemeral `xadd` (the quick win) — ship and soak first, zero interaction with the gate                                                                         | Yes                                         |
| 2        | Add `runActive`/`activeRunId` to `projectThreadHistory` + `AgUiHistoryMessagesResult`; parse client-side but do NOT change the gate; assert in the harness that `runActive` matches durable run-context across recorded runs | Yes (nothing reads it for the decision yet) |
| 3        | Flip the gate behind a feature flag, keeping `isAgentWorking` as a secondary close hint; roll out via the existing `historyLoadKey` remount seam                                                                             | Flag-off instantly reverts                  |
| 4        | Collapse the connect-time double Postgres read (`?history=messages` GET re-reads rows the SSE replay already emits) — cursor-only or seed from SSE replay                                                                    | Yes                                         |
| Deferred | The `O(viewers × duration)` `xread` long-poll fan-out cost — a transport rewrite (PartyKit/`apps/broadcast` or shared Redis pub/sub). Explicitly out of scope                                                                | —                                           |

**Risk.** Medium. The dangerous inverse failure mode: `runActive` computed stale-TRUE plus a server that never converges on terminal could hold a 300s invocation open. Mitigation: keep `isAgentWorking` as a secondary close hint and add a span at each `console.warn` swallow so terminal-lag is observable before the change makes it costlier. Set `MAXLEN N` generously (resume window << N); worst case on a slow reconnect is a redundant fuller DB replay, not data loss. Respects the pinned-version constraint — changes only what is passed into `unstable_resume`, a documented input.

### Bet B — Proposal 1 (Changes B+C): Typed status state machine + optimistic-overlay registry

**Problem.** The reducer encodes its most important invariants in convention and tests rather than types. `runStarted` is derived four different ways across four files (`status-setter.ts:27`, `lifecycle-events.ts:135,174`, `snapshot-adapter.ts:166`), so illegal sequences like stale-complete→booting are handled by ad-hoc preserve flags rather than rejected. Five optimistic flags (`hasLiveLifecycleEvents`, `hasOptimisticUserSubmit`, `hasOptimisticQueuedMessages`, `hasOptimisticPermissionMode`, plus the `optimisticSubmission` stash) drive a 75-line per-field `shouldPreserveX` matrix in `applySnapshot` (`reducer.ts:141-217`); `isSnapshotNoOp` re-enumerates the same 12 fields. Each new optimistic concern adds a flag AND an `applySnapshot` branch AND an `isSnapshotNoOp` branch AND a clear-rule.

**Proposed change.**

- **Change B:** Add `status-machine.ts` — a pure `transition(current, input) → next` over a closed input union (optimistic-submit, run-started, run-finished, run-error, status-changed, durable-adopt). `runStarted` is derived in ONE place. All four current derivation sites call `transition`. The `durable-adopt` input encodes the stale-complete→booting rule explicitly and rejects the regression.
- **Change C:** Replace the five flags + the `applySnapshot` matrix with a single nullable `optimisticOverlay` tagged union (`user-submit | queued | permission-mode`). State splits into a durable base (filled only from snapshots) plus the overlay. `projectThreadViewModel` composes base+overlay; `applySnapshot` then ONLY replaces the base, so the whole preserve matrix and the `isSnapshotNoOp` short-circuit disappear. Coupled with C, eliminate `pendingClientSubmissionIdRef`: the overlay holds `clientSubmissionId`, so expose a `pendingClientSubmissionId` selector that `chat-ui` reads in `onAppendRejected`.

**Migration phases.** (Change A — the dual-field collapse — is the quick win and lands first.)

| Phase | Change                                                                                                                                                                                                                                                                                                         |
| ----- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 0     | Characterization: a transition-table test capturing current `status`/`runStarted` outputs for every `currentStatus × event` pair the four derivations produce. Record the green baseline. No production change.                                                                                                |
| B     | Introduce `transition`; reroute `applyLifecycleEvent`, `applyLifecycleCustomEvent`, the optimistic flip, and snapshot adoption through it. Phase-0 table is the acceptance gate — outputs must match unless a divergence is an intentional, called-out fix.                                                    |
| C     | Replace the five fields with `optimisticOverlay`; split `applySnapshot` into base-replace + overlay-compose; rewrite `optimistic-events`; keep the `clientSubmissionId`-keyed bubble suppression the duplicate-replay-bubble tests accept (`reducer.test.ts:183`). Land separately so a bisect can isolate it. |

**Risk.** Medium. The reducer is the load-bearing seam for the resume deadlock: `lifecycle.threadStatus → effectiveThreadStatus → isAgentCurrentlyWorking → resolveRuntimeResumePolicy → whether the SSE opens. A transition bug that flips status at the wrong moment could regress the deadlock or open spurious subscriptions. Mitigation: Phase-0 characterization test must pin exact current outputs BEFORE refactoring; run the integration replayer at every phase boundary. The no-op guard is reference-equality fragile today, so the base/overlay split must keep an equivalent no-op fast path or every value-equal snapshot rebuild churns the reducer.

### Bet C — Proposal 4 (scoped to Phases 1-3): One typed ChatErrorEvent end-to-end

**Problem.** Chat errors travel two disconnected channels. The live runtime collapses 4+ distinct failures into one `onError(error: Error)` disambiguated ENTIRELY by regex on `error.message` (`/Run already in progress/i` for lock-held; two literal `verifyEvents` strings). The server emits `RUN_ERROR` WITH a typed `code`, but `@ag-ui/client`'s `onRunFailed({error: Error})` collapses it to a bare `Error` and the `code` never reaches the UI. Worse, three different enum spaces derive from one daemon failure (`DaemonTerminalErrorCategory` 5 values, `ThreadErrorType` 18 values, `RuntimeFailureCategory` 17 values), and `ChatError`'s switch is keyed on `ThreadErrorType` only — so a live-only error always falls to the generic "An error occurred" default. `ChatError`'s exhaustiveness is fake: `chat-error.tsx:427-430` is a `console.log(_exhaustiveCheck)` with `// TODO: make typescript error here`, so a new `ThreadErrorType` compiles clean and renders the generic fallback.

**Proposed change.** One typed `ChatErrorEvent` discriminated union in shared. Align the `RUN_ERROR` `code` from `DaemonTerminalErrorCategory` to the same `ThreadErrorType` the DB persists (one `deriveChatFailure()` consumed by both the AG-UI mapper and the DB persistence path). Split the client `onError` seam into `onRunFailed`/`onAppendRejected`/`onTransportError` (the runtime already distinguishes these internally; `react-ag-ui`'s subscriber already reads `error.code`). Make `ChatError`'s `default` a real `const _: never = code` compile error and switch on the typed `code`.

**Migration phases (scoped).**

| Phase | Change                                                                                                                                                                                                                                                                                                    |
| ----- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 0     | Define `ChatErrorEvent` + failure span names in shared; add `deriveChatFailure()`, have both `buildRunTerminalAgUi` and `buildRunContextFailureUpdates` call it, asserting today's values. Pure refactor. (This is the quick-win slice.)                                                                  |
| 1     | Server, additive: change `RUN_ERROR` `code` (`daemon-event/route.ts:969`, `ag-ui-mapper.ts`) to the `ThreadErrorType`-aligned code AND stamp `clientSubmissionId`/`runId` on `rawEvent`. Client still ignores `code`. Verify via recorder/replayer that the code matches the persisted `ThreadErrorType`. |
| 2     | Client, additive: wrap `useAgUiRuntime` to expose the three typed callbacks; `handleRuntimeError` reads the typed code when present, FALLS BACK to the existing regex when absent. Both paths coexist; no deletion.                                                                                       |
| 3     | UI: `ChatError` switches on the typed `code`; convert the `default` to a never-compile error; extract `pickActiveChatError` and remove the duplicated precedence from `chat-ui-layout.tsx:140-142` and `resolveThreadRuntimeErrorProps`. Fix any newly-surfaced exhaustiveness gaps.                      |

Observability/span plumbing (the original P4 Phase 5) is a separable follow-on, not a blocker.

**Risk.** Medium, concentrated in Phase 2. The split relies on `react-ag-ui@0.0.26` surfacing `onRunFailed` and `@ag-ui/client@0.0.52` not stripping `error.code`; keep the wrapper defensive (fall back to regex if a typed field is absent) and assert the boundary in the harness so a future bump fails a test, not prod. Compile-exhaustiveness will surface currently-unhandled `ThreadErrorType` cases at build time — that is the point, but budget for handling them before merge.

## Full proposal detail

### Proposal 1 — Typed status state machine + optimistic-overlay registry

Score v7-8/r5/e5. The strongest maintainability bet. Change A (dual-field collapse) is a quick win; Changes B+C are Bet B above. Converts the highest-churn, test-enforced invariants into compile-time facts and collapses every future optimistic concern from "flag + `applySnapshot` branch + `isSnapshotNoOp` branch + clear-rule" down to one union arm plus one clear rule. Strictly better-scoped than Proposal 5 for the same defects because it does NOT delete the transcript copy. The `clientSubmissionId`-on-error-payload sub-change (Phase 5 in the original) crosses into the error subsystem — defer it to the shared wire-contract change.

### Proposal 2 — Server-authoritative subscription + bounded Redis cost

Score v8/r6/e5-6. Bet A above. Two genuinely separable pieces bundled in the original: a trivial-risk Redis buffer fix (the quick win) and a higher-risk subscription-gate flip. They have zero interaction and must be separated. The buffer fix ships now; the gate flip is the flagged strategic change. Eliminates the gated-on-self deadlock CLASS rather than patching individual mis-derivations, and decouples transport liveness from optimistic-submit semantics (today the optimistic "booting" flip is load-bearing for opening the stream at all). The `O(viewers × duration)` long-poll fan-out cost is correctly left out of scope.

### Proposal 3 — Collapse the dual follow-up write path onto runtime.append

Score v7/r5-6/e6. One follow-up is written two structurally different ways (runtime `append` → AG-UI POST → decode-back → `followUpInternal`, vs `publish({type:'send-message'})` → `followUp()` server action → `followUpInternal`) with NON-equivalent server semantics: the append path runs inside `withFollowUpSubmissionGuard` (dedupe + run-lock) while the fallback `followUp()` has neither. The Phase-1 slice — wrapping the fallback in the guard — is a quick win that closes the idempotency hole immediately. The full collapse (route-enum classify-once, deleting the `send-message` intent variant, typing the lock-held error instead of `/Run already in progress/i` regex) is a roughly one-engineer-week effort whose hard part is updating the 29-case routing test plus three integration replay tests. Sequence after P4 lands the typed `error.kind` and the `clientSubmissionId` echo, which this proposal consumes.

### Proposal 4 — One typed ChatErrorEvent end-to-end

Score v7-8/r5-6/e6-7. Bet C above. Highest user-facing value: live run-failures currently always render the generic "An error occurred" banner (confirmed). It is the load-bearing dependency that unblocks data-correlated optimistic rollback — carrying `clientSubmissionId` on the typed error payload removes the single-in-flight `pendingClientSubmissionIdRef` constraint that Proposals 1, 3, and 5 all separately want. Aligning the `RUN_ERROR` code to `ThreadErrorType` and replacing message-regex classification with typed data directly defuses the pinned-version churn risk the codebase already flags. Sequence the server enum-alignment (Phases 1-2) early because others depend on it.

### Proposal 5 — Durable event log as single transcript authority; demote reducer to sidecar

Score v6-7/r7-8/e8. The highest-effort, highest-risk proposal, and substantially a re-bundling of P1+P3 plus one unique piece: deleting the reducer's `dbMessages` copy. Do NOT fund it as an independent epic. P5's phases 1-2-4 (status collapse, overlay, ref removal) ARE P1's A-B-C. Its unique value — the durable AG-UI event log as sole transcript authority and deleting the redundant `dbMessages`/`sidePanel.messages` copy — is real but lower-risk than P5's 2-3 PR framing implies, because that copy already has no render consumer (the visible transcript renders from the assistant-ui runtime; SecondaryPanel and redo/fork read the raw snapshot, not the reducer's projected `dbMessages`). Recommendation: ship P1, then treat the `dbMessages` deletion as a separate small optional follow-up gated on an equivalence test, rather than dragging the status/overlay work behind a transcript-authority epic.

## Sequencing

**Do first (parallel, independent, low-risk):**

1. Redis `MAXLEN` + `EXPIRE` (P2 Phase 1) — immediate cost bleed-stop, zero interaction with anything.
2. Dual-status-field collapse (P1 Change A) — half a day, makes an invariant type-impossible.
3. Guard the fallback `followUp()` (P3 Phase 1 slice) — closes a real idempotency hole.
4. P4 Phase 0 + the three live-tail swallow spans — additive, makes the swallow class countable before riskier work.

**Then, the one blocking dependency:** P4 Phases 1-2 (server `RUN_ERROR` code → `ThreadErrorType`, stamp `clientSubmissionId`/`runId` on `rawEvent`, wrap the client `onError` seam). This single wire-contract change is the prerequisite for data-correlated rollback in P1, P3, and P5. Land it once.

**Then, the two strategic spines (stagger, do not parallelize on the fresh code):**

- P2 Phases 2-3 (server-authoritative liveness, flagged) — defuses the deadlock class.
- P1 Phases B-C (status machine + overlay) — gated on the Phase-0 characterization test.

**Last, optional:** P3 full routing collapse (consumes P4's typed error), then P5's `dbMessages` deletion (consumes P1's overlay). Both decomposable, neither monolithic.

**Why this order:** the quick wins are independent and bank value immediately. P4's server enum-alignment is the shared unblock four proposals want, so it goes before them. P2-gate and P1-BC both touch the just-shipped optimistic-submit path and the recently-fixed resume gate; staggering them and gating each on the integration replayer (`apps/www/test/integration/`) keeps blast radius isolated and bisectable.
