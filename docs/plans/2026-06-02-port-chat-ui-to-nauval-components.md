# Port Terragon chat UI to nauvalazhar/ai components (keep AG-UI transport)

Status: proposed
Date: 2026-06-02
Owner: chat layer

## Summary & recommendation

**Recommendation: keep the assistant-ui runtime as the data layer and skin its leaf renderers with vendored nauval components. Do NOT drop the runtime. But re-target the work onto the surface that is actually live.**

The big correction since the first draft: the live agent transcript is `assistant-ui/native-thread.tsx`, NOT the `PART_REGISTRY` / `TOOL_DISPATCH` / legacy `ChatMessage` agent branch. Verified topology (see "Live vs. dead surfaces" below): `chat-ui.tsx` → `terragon-thread-runtime-content.tsx` → `terragon-transcript-surface.tsx` mounts `<NativeThread />` (runtime parts: text / reasoning / tool-call) **plus** `lifecycleMessages` → `TerragonSystemMessage` → `ChatMessage`, where `ChatMessage` **short-circuits to `SystemMessage` for `role === "system"` (chat-message.tsx:99-111)** and never reaches `MessagePart` / `PART_REGISTRY` / `TOOL_DISPATCH` for any live agent message. Live tool calls, reasoning, and text are rendered by the hand-rolled `NativeText` / `NativeReasoning` / `NativeToolCall` / `NativeToolGroup` inside `native-thread.tsx`, which touch none of the dispatch tables. The only live consumers of the `PART_REGISTRY`/`SystemMessage` path are **git-diff cards and system/lifecycle notices**.

That correction drives three structural changes from the first draft:

1. **Highest-leverage target is `native-thread.tsx`, not `tools/generic-ui.tsx`.** Re-skinning `generic-ui.tsx` changes how tools look _inside system/legacy cards_, which is not what the user sees in the transcript. The live tool/reasoning UX is `NativeToolCall`/`NativeReasoning`. They are skinned first.
2. **The `PART_REGISTRY` / `TOOL_DISPATCH` / legacy-`ChatMessage`-agent-branch skin is OUT OF SCOPE for this port until the consolidation owner confirms the surface is staying.** AGENTS.md and `docs/plans/2026-04-30-runtime-owns-writes-adr.md` describe an in-flight consolidation onto the assistant-ui runtime owning rendering; the live transcript already moved to `NativeThread`. Investing six phases into skinning a surface the consolidation may retire is wasted polish. Phase 0 includes an explicit gate question; if the answer is "retiring," the per-tool and rich-part `PART_REGISTRY` rows are dropped from scope and only `NativeThread` + the live system/git-diff surfaces are skinned.
3. **Three live system-message surfaces that the first draft omitted are now mapped:** the live `git-diff` card subtree (`git-diff-*.tsx`, including the load-bearing `CommentWidget` injection), the `SystemMessage` `message_type` family (`chat-message-system.tsx`, 12 variants), and the meta-chips' **second** host `chat-prompt-box.tsx`.

What stays untouched, verified: AG-UI transport and the `routeComposerSubmit` → `runtime.append` → `handleAgUiPostCommand` → `dispatchFollowUpFromAppend` → `followUp()` writer pipeline; the cancel path; the `useScrollToBottom` autoscroll engine (which lives in **`chat-ui.tsx`**, not `chat-ui-layout.tsx`); streamdown + `parseIncompleteMarkdown`; TipTap as composer value-owner; the dispatch tables and their `Assert<>` exhaustiveness guards. nauval is store-less and transport-less — every nauval component takes its visual `state` as a string prop the call site computes — so it slots cleanly under the existing seams as a pure presentational skin.

**`NativeToolCall` → rich-tool decision (resolves the biggest hidden-coupling risk).** `NativeToolCall` is a `ToolCallMessagePartComponent` fed by runtime props; it does **not** flow through `buildProps`/`TOOL_DISPATCH`, so the rich per-tool renderers (`bash-tool`, `read-tool`, `file-change-tool`, …) are **not reachable** from the live transcript. Two options:

- **(A) Generic nauval `Tool` skin over the existing `<details>`** — cheap, no per-tool richness. Live transcript keeps a uniform tool card (name + args + raw result), just nicer chrome. **CHOSEN for this port.**
- **(B) Route runtime tool parts through `TOOL_DISPATCH`** to get bash exit codes / ANSI / inline diffs in the live transcript — this is a **real architecture change** (new adapter from runtime `tool-call` props into `buildProps` input shape), not a re-skin. **Explicitly out of scope** here; if product wants per-tool richness in the native transcript, file it as a separate workstream.

Table (b) below maps the per-tool renderers for reference only; under option (A) **none of them reach the live transcript** and they are skinned only if the consolidation gate keeps the `TOOL_DISPATCH` surface.

## Vendoring strategy

nauval has no npm package and no runtime — components are shadcn-style copy-in. Vendor them into the repo.

1. **Directory.** `apps/www/src/components/ai/` (one file per nauval component actually used by this port: `tool.tsx`, `conversation.tsx`, `message.tsx`, `reasoning.tsx`, `confirmation.tsx`, `console.tsx`, `composer.tsx`, `status.tsx`, `action.tsx`, `loader.tsx`, `source.tsx`, `player.tsx`, `attachment.tsx`, `uploader.tsx`, `usage-meter.tsx`, `callout.tsx`, plus `agent-run.tsx`/`todo.tsx`/`diff.tsx`/`code-block.tsx`/`file-tree.tsx`/`generated-image.tsx`/`markdown.tsx` **only if** the consolidation gate keeps the `PART_REGISTRY` surface or a later panel slice adopts them). Keep vendored files unmodified except import-alias + token rewrites — Terragon adaptations live in the leaf renderers, not the vendored files.
2. **How to bring them in.** Prefer the CLI per component (`npx @nauvalazhar/ai@latest add tool conversation message ...`) targeting alias `@/components/ai`, which rewrites imports automatically; otherwise copy each `src/components/ai/<name>.tsx` from `github.com/nauvalazhar/ai`. **Exact-pin every new dep** in `apps/www/package.json`; do not loosen the existing `@assistant-ui/react@0.12.24`, `@assistant-ui/react-ag-ui@0.0.26`, `@ag-ui/{core,client}@0.0.52` pins.
3. **Base UI is a NET-NEW second primitive stack — this is a risk, not a routine dep line.** Verified: the repo has **zero** `@base-ui` packages today and 18 `@radix-ui` packages (shadcn/ui's primitive layer). nauval primitives (Collapsible/Popover/ScrollArea/Meter/Tooltip/Switch) sit on Base UI. **Decision required before Phase 0 completes:** either (a) accept Base UI alongside Radix as a second stack (bundle + maintenance cost, possible a11y/behavior divergence between nauval components and the rest of the app), or (b) retarget the vendored nauval primitives onto the existing Radix primitives during vendoring. Recommendation: for the small live-surface set this port touches (`Tool`/`Reasoning`/`Message`/`Status`/`Console`/`Composer` chrome), audit which actually pull Base UI; retarget the cheap ones onto Radix and accept Base UI only where retargeting is expensive. Record the decision in `VENDOR.md`. Do not treat this as "exact-pin and move on."
4. **Keeping them updatable.** Record the nauval commit SHA per file in `apps/www/src/components/ai/VENDOR.md` (component → upstream path → SHA → date → "Base UI or retargeted-to-Radix"). Re-running `add` regenerates from upstream; because Terragon logic lives in the leaf wrappers, regeneration is a clean overwrite + re-diff of alias/token/primitive rewrites only.
5. **Theming alignment (Tailwind v4, `^4.1.10`).** Map nauval's semantic tokens (`bg-surface`, `bg-surface-elevated`, `border-border`, `text-muted-foreground`, `text-foreground`, `bg-primary`, `text-destructive`, custom `inflight` accent) to Terragon's existing `@theme` tokens in the global stylesheet rather than hardcoding colors. Alias nauval `surface`/`inflight` to the existing `--shadow-warm-lift`/cream-card palette so bubbles match today's look. Define diff tokens `--color-diff-added`/`--color-diff-removed` in `:root` and `.dark` only if a diff component is ever adopted (not in the base scope, since git-diff stays and `diff-part` is dead). `markdown.tsx` is **not adopted** (streamdown stays), so skip `@plugin "@tailwindcss/typography"`.

## Risks & open questions

1. **Topology was inverted in the first draft (now corrected).** The live transcript is `native-thread.tsx`, not `PART_REGISTRY`/`TOOL_DISPATCH`. Phases now target the live surface; dead surfaces are gated behind Phase 0 / Phase 7. This is the single most important correction.
2. **Consolidation may retire the `PART_REGISTRY`/`TOOL_DISPATCH` surface (blocking gate).** Resolved by the Phase 0 gate question. Do not skin tables (b)/(c) until answered.
3. **`NativeToolCall` rich-tool richness is NOT free.** Option A (chosen) gives a generic nauval `Tool` skin; bash exit codes / ANSI / inline diffs do **not** reach the live transcript without option B (a real architecture change routing runtime parts through `TOOL_DISPATCH`). Out of scope; flag to product if richness is wanted.
4. **Streaming markdown UX (high risk).** nauval `Markdown` lacks `parseIncompleteMarkdown` + block segmentation; adopting it regresses streaming and reopens the code-highlight bundle problem. **Decision: keep streamdown; nauval supplies typography tokens only.** (Note: the rejected `@assistant-ui/react-markdown` comparison from AGENTS.md is theoretical — that package is not installed, so there is nothing to "remove.")
5. **Scroll ownership (and the file that owns it).** The scroll engine is in **`chat-ui.tsx`** (`useScrollToBottom`/`isAtBottom`/`forceScrollToBottom`/`messagesEndRef`, verified `:160`), NOT `chat-ui-layout.tsx`. nauval `Conversation` must be a dumb frame; its scroll button wires to our existing props. Edit the right file in Phase 1.
6. **Base UI as a second primitive stack (decision required).** Net-new alongside 18 Radix packages. Resolve in Phase 0 (accept vs retarget onto Radix); record in `VENDOR.md`.
7. **Composer submission/queueing ownership.** nauval `Composer` defaults to owning text + `onSubmit` + clear; TipTap owns the value and `routeComposerSubmit` owns append-vs-queue. **Decision: controlled chrome only; nauval never constructs/sends/queues a message.** Phase-5 ordering resolved: independent of the ADR's `ComposerPrimitive.Root` wrap; co-landing not required.
8. **git-diff inline comments (resolved, not open).** The live comment widget lives in `git-diff-file-wrapper.internal.tsx` (`CommentWidget` via `DiffLineAnnotation<CommentWidgetData>`), which nauval `Diff` cannot host. **Decision: keep the entire `git-diff-*` subtree; do not route it through nauval `Diff`.** The separate `diff-part.tsx` `DiffPartView` is a plain `<pre>` with **no** comment slot (verified `:136`) and is DEAD anyway — the first draft's "diff has no comment slot" risk was attached to the wrong component.
9. **Exhaustiveness-assertion preservation.** The `Assert<>` guards in `part-registry.ts` (~277-290), the typed `TOOL_DISPATCH` map, and the two `never` checks in `chat-message-system.tsx` (`:60`, `:86`) break the build if a key is added/removed. A pure re-skin only repoints component bodies, so the guards stay green. **Risk: changing a leaf's `React.ComponentProps` shape without updating `buildProps` breaks `tsc-check`.** Mitigation: keep each leaf's prop shape equal to what its caller already passes; never edit `buildProps`, registry keys, or the `never`-guarded switches.
10. **Story/test churn.** Class-based assertions (`toHaveClass`) churn on a re-skin; prefer role/text assertions where tests touch them. Tool tests live under `chat/tools/` — use correct paths.
11. **Accessibility deltas.** nauval emits its own `data-*`/aria. **Keep custom a11y where nauval has no equivalent** (the `role="progressbar"` TodoWrite checklist and `role="treeitem"` file trees are on the DEAD path; the live a11y to protect is the secondary-panel tab keyboard nav and `Confirmation` button accessible names if/when the dead surface is skinned).
